# Site Engine - Prototype Reference

> Source: `ai-site-builder/master` branch
> Purpose: Visual website builder using AI to create/edit page blocks
> Stack: Vanilla Node.js HTTP server, MySQL, React 18, esbuild, OpenAI + Flux 2 Pro

This document captures every proven pattern from the Site Builder prototype. The new implementation will be built from scratch on `@ab/app-foundation` but must replicate these patterns.

---

## Architecture Summary

The Site Builder is a JSON-first visual page builder. Sites are stored as a tree of typed "blocks" in a JSON column. The editor renders blocks with interactive overlays (hover, select, drag-and-drop). AI modifies the block tree via chat, and a publishing pipeline converts the JSON to static HTML/CSS.

**Core flow**: User chats → AI modifies JSON block tree → Editor re-renders → User publishes → Static HTML/CSS generated

## Block Schema

### Block Structure

Every block has this shape:

```javascript
{
  id: 'container-1700000000-abc123',  // {kind}-{timestamp}-{random}
  kind: 'container' | 'text' | 'image' | 'icon' | 'link' | 'list' | 'code' | 'custom',
  tag: 'div',                          // HTML tag to render
  content: 'Hello world',             // Text content (for text/link blocks)
  src: 'https://...',                  // Image source (for image blocks)
  alt: 'Description',                 // Image alt text
  href: 'https://...',                // Link target
  iconName: 'Star',                   // Lucide icon name (for icon blocks)
  wrapper: { /* CSS properties */ },   // Outer wrapper styles (spacing, positioning)
  inner: { /* CSS properties */ },     // Inner element styles (colors, typography)
  children: [ /* nested blocks */ ],   // Child blocks (for containers)
}
```

### Block Kinds

| Kind | Purpose | Key Properties |
|------|---------|---------------|
| `container` | Layout wrapper | `children[]`, `wrapper`, `inner` |
| `text` | Text content | `content`, `tag` (h1-h6, p, span) |
| `image` | Images | `src`, `alt`, `wrapper`, `inner` |
| `icon` | Lucide icons | `iconName`, `wrapper`, `inner` |
| `link` | Hyperlinks | `content`, `href`, `tag` (a) |
| `list` | Lists | `children[]` (each child is a list item) |
| `code` | Code blocks | `content` |
| `custom` | Raw HTML | `content` (treated as raw HTML) |
| `app` | Embedded app | `appId`, `appUrl`, `appSlug`, `height` (iframe embed of a standalone app) |

### Block ID Format

```
{kind}-{timestamp}-{random}
// Example: container-1700000000-x7k2m
```

Generated by `ensureIds()` on every save. IDs are stable once assigned.

### App Block (Embedded Apps)

The `app` block kind allows embedding standalone apps inside a site. This enables the Site Builder to incorporate dynamic features (booking widgets, calculators, forms with databases, etc.) that go beyond static content.

```javascript
{
  id: 'app-1700000000-x7k2m',
  kind: 'app',
  appId: 'scheduler',               // app registry ID (pre-built or custom)
  appSlug: 'scheduler',             // for URL resolution
  appUrl: '/apps/scheduler/public/book/acme-salon', // specific view to embed
  height: '600px',                  // iframe height (width fills wrapper)
  wrapper: { /* spacing, positioning */ },
  inner: { /* border, background, overflow: hidden, border-radius */ },
}
```

**Rendering**: The editor and published output render this as an `<iframe>` pointed at `appUrl`. The wrapper controls the block's position in the page layout, the inner provides visual framing (border-radius, shadow, etc.), and the iframe fills the inner space.

**How apps get embedded**:
1. **Existing app**: User says "add a booking widget" in Site Builder chat. AI recognises this maps to the Scheduler's public booking page, creates an `app` block with the correct `appUrl`
2. **New app on-the-fly**: User says "add a mortgage calculator". AI determines no existing app matches, spawns an App Builder session to create it as a custom app, then embeds it as an `app` block once the custom app is ready

**In the editor**: The iframe loads the app's live URL. The wrapper/inner overlays still appear for selection, drag-and-drop, and the hover menu (same as other blocks). The app inside the iframe is interactive but contained.

**On publish**: The iframe `src` points to the production URL of the embedded app. Same-origin policy is satisfied since all apps share the same domain via path routing.

### Wrapper/Inner CSS Split Pattern

**Critical architecture decision**: Every block has two style objects.

```javascript
// common/styles.js - buildWrapperStyle() / buildInnerStyle()
// 112 lines

// WRAPPER: Controls positioning, spacing, layout behavior
buildWrapperStyle(block) {
  // margin, padding, width, height, position, display, flex properties,
  // grid properties, overflow, z-index, box-sizing
}

// INNER: Controls visual appearance
buildInnerStyle(block) {
  // background, color, font-*, text-*, border, border-radius, box-shadow,
  // opacity, transform, transition, cursor, list-style
}
```

**Why two layers?**
- Editor artifacts (selection borders, hover outlines, context menus) attach to the wrapper
- Inner styles are the actual visual appearance (including complex layouts like grid, flexbox)
- Adding editor artifacts directly to the inner element would break complex layouts (e.g. a border on a flex container changes its dimensions, an overlay inside a grid disrupts grid flow)
- Published output should **not** flatten both into a single element - complex layouts will break. The wrapper/inner structure must be preserved in published output too

## Site Data Model

### Database Schema (MySQL)

```sql
-- sites table
CREATE TABLE sites (
  id INT AUTO_INCREMENT PRIMARY KEY,
  user_id INT NOT NULL,
  name VARCHAR(255) NOT NULL,
  slug VARCHAR(255) NOT NULL UNIQUE,
  current_json LONGTEXT,           -- Current block tree JSON
  references_json LONGTEXT,         -- AI reference data (colors, fonts, etc.)
  published_version INT DEFAULT 0,
  status ENUM('draft', 'published') DEFAULT 'draft',
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

-- site_versions table (version history)
CREATE TABLE site_versions (
  id INT AUTO_INCREMENT PRIMARY KEY,
  site_id INT NOT NULL,
  version_number INT NOT NULL,
  json_snapshot LONGTEXT,          -- Full block tree snapshot
  html TEXT,                       -- Generated HTML
  css TEXT,                        -- Generated CSS
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (site_id) REFERENCES sites(id)
);

-- media_assets table
CREATE TABLE media_assets (
  id INT AUTO_INCREMENT PRIMARY KEY,
  site_id INT NOT NULL,
  original_url TEXT,
  local_path VARCHAR(512),
  asset_type ENUM('image', 'icon', 'font'),
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (site_id) REFERENCES sites(id)
);
```

## Image Generation (Flux 2 Pro)

### API Integration (`services/images.js`)

```javascript
// images.js - 125 lines

const BFL_API_URL = 'https://api.bfl.ai/v1/flux-2-pro';

async function generateImage(prompt) {
  // 1. Submit generation request
  const { id: requestId } = await fetch(BFL_API_URL, {
    method: 'POST',
    headers: { 'X-Key': BFL_API_KEY },
    body: JSON.stringify({
      prompt: prompt + PHOTOREALISTIC_SUFFIX,
      width: 1024,
      height: 768,
    })
  });

  // 2. Poll for completion (3s interval, 60s timeout)
  while (true) {
    const status = await fetch(`https://api.bfl.ai/v1/get_result?id=${requestId}`);
    if (status.status === 'Ready') return status.result.sample;
    if (status.status === 'Error') throw new Error('Generation failed');
    await sleep(3000);
  }
}
```

### History/Undo (`index.jsx`)

```javascript
// Snapshot-based undo (max 100 snapshots)
// Stored in localStorage for persistence across refreshes

const MAX_HISTORY = 100;
let history = [];
let historyIndex = -1;

function pushHistory(snapshot) {
  // Truncate future if we've undone some steps
  history = history.slice(0, historyIndex + 1);
  history.push(JSON.stringify(snapshot));
  if (history.length > MAX_HISTORY) history.shift();
  historyIndex = history.length - 1;
}

function undo() {
  if (historyIndex > 0) {
    historyIndex--;
    return JSON.parse(history[historyIndex]);
  }
}

function redo() {
  if (historyIndex < history.length - 1) {
    historyIndex++;
    return JSON.parse(history[historyIndex]);
  }
}

// Ctrl+Z / Ctrl+Y keyboard shortcuts
// Cmd+Z / Cmd+Shift+Z on Mac
```

## Publishing Pipeline (`services/sites.js`)

### `renderStaticAssets(siteJson, referencesJson)`

Converts JSON block tree to static HTML + CSS:

```javascript
function renderStaticAssets(siteJson, referencesJson) {
  // 1. Walk the block tree
  // 2. Generate HTML elements preserving wrapper/inner structure
  //    - Wrapper div with positioning/spacing styles
  //    - Inner element with visual/layout styles (grid, flexbox, etc.)
  //    - Convert React-style camelCase to CSS kebab-case
  // 3. Extract common styles into CSS classes
  // 4. Minify CSS
  // 5. Generate complete HTML document with:
  //    - DOCTYPE, html, head (meta, title, CSS), body
  //    - Google Fonts links from references
  //    - Responsive viewport meta tag
  // 6. Return { html, css }
}
```

### Version Management

```javascript
async function publish(siteId) {
  const site = await getSite(siteId);
  const { html, css } = renderStaticAssets(site.currentJson, site.referencesJson);

  // Create new version
  const versionNumber = site.publishedVersion + 1;
  await createVersion(siteId, versionNumber, site.currentJson, html, css);

  // Update site
  await updateSite(siteId, { publishedVersion: versionNumber, status: 'published' });

  // Keep last 10 versions, delete older ones
  await pruneOldVersions(siteId, 10);
}
```
